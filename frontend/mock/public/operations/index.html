Operations design note運用設計ノート

This note is design intent, not an operations report. Nothing described below is built or running: there is no deployment, no alert has ever fired, no dashboard exists, and no on-call rotation is in place. The product this note would operate does exist, and where one of its shipped behaviors is load-bearing for the design, the note marks it built; everything else is design — future work, recorded now so the eventual build has a target it can be checked against.

ここに書いてあるのは運用の設計であって、運用した記録ではありません。以下のものは、まだ何ひとつ作られておらず、動いてもいません。デプロイは存在せず、アラートが鳴ったことは一度もなく、ダッシュボードもオンコール体制もまだありません。運用の対象になる製品そのものは存在するので、出荷済みの挙動が設計を支えている箇所には built の印を付けています。それ以外はすべて design — まだ作っていない将来のものを、いざ作るときに照らし合わせられる目標として、いまのうちに書き残しておくものです。

Why write this before building itなぜ作る前に書くのか

An agent that operates a legacy system has an unusual failure profile: the dangerous failures do not crash. They submit. A model that has started flailing produces well-formed keystrokes that pass through the adapter exactly as faithfully as correct ones, and ordinary error logging — built to catch exceptions — sees nothing wrong. Operating such a system is therefore less about uptime and more about noticing wrongness early, diagnosing it fast, and bounding what a wrong action can cost. That conviction already shaped the product: every executed step asserts the resulting screen state, failures roll back, and nothing executes without a human approval. This note extends the same conviction past the product's edge, to the layer that would watch it day after day.

レガシーシステムを操作するエージェントの故障の形は独特です。危険な故障はクラッシュしません — 送信されてしまうのです。挙動の崩れたモデルが整形式のキー入力を生成すれば、アダプタは正しい入力と同じ忠実さでそれを流し、例外を捕まえるために作られた通常のエラーロギングには何の異常も映りません。したがってこのシステムの運用は、稼働率の話である以上に、「誤りに早く気づくこと」「速く診断すること」「誤った操作の被害を限定すること」の話になります。この確信は製品自体をすでに形づくっています — 実行された各ステップは画面状態を検証し、失敗はロールバックし、人の承認なしには何も実行されません。本ノートは同じ確信を製品の縁の外、つまり日々それを見守る層へと延長するものです。

The three-layer funnel3層の漏斗

design Operations is structured as a funnel of three layers, ordered by how the operator encounters a problem: first you are called (alerts), then you see (a dashboard), then you ask why (drill-down). The deliberate principle behind the funnel: operations is not a separate system bolted on later, but a reader of records the product already writes — task steps, approval decisions, verify outcomes, the audit log — plus one instrumentation point that is itself still unbuilt: a wrapper around the LLM client logging tokens, cost, latency, retries and model per call.

design 運用は3層の漏斗として設計します。並び順は、運用者が問題に出会う順序そのものです — まず呼ばれる（アラート）、つぎに見る（ダッシュボード）、最後になぜかを問う（ドリルダウン）。漏斗の背後にある意図的な原則：運用は後付けの別システムではなく、製品がもともと書き残す記録 — タスクステップ・承認の決定・verify の結果・監査ログ — の読み手である、ということです。これに、それ自体まだ未構築の計測点が一つ加わります。LLM クライアントを包み、呼び出しごとにトークン・コスト・レイテンシ・再試行・モデルを記録するラッパーです。

Layer 1 — alerts: being called第1層 — アラート：呼ばれる

design The design calls for four families of alert rules, each tied to a distinct failure mode:

design 設計しているアラートルールは4系統で、それぞれ別の故障モードを受け持ちます。

  • Heartbeat death — a health endpoint stops answering: the plain outage case.ハートビートの死 — ヘルスエンドポイントが応答しなくなる、素朴な停止のケース。
  • Verify-failure-rate spike — the screen keeps disagreeing with the plan. This is the tripwire for silent malfunction: the system has started doing the wrong thing without crashing.verify 失敗率の急騰 — 画面が計画と食い違い続ける。クラッシュなしに誤った動作が始まったことを捕まえる、静かな誤作動への仕掛け線です。
  • Error / retry-rate spike — the adapter or the legacy target has changed under the agent's feet.エラー／再試行率の急騰 — アダプタかレガシー側が、エージェントの足元で変わったことの兆候。
  • Token / cost surge with a budget cap — the runaway-loop safeguard: calls per task climbing past the budget means the loop is spinning, not working.トークン／コストの急増と予算上限 — 暴走ループへの安全装置。タスクあたりの呼び出しが予算を超えて伸びるのは、働いているのではなく空回りしている兆候です。

The designed channel is the same messenger the assistant itself uses, so the operator's phone is the pager — an alert like "verify failure rate is climbing" arrives as a push message. Thresholds are deliberately not stated in this note: inventing them on paper would be exactly the kind of unmeasured claim the rest of this project avoids. They would be set from observed baselines during a measurement period, then tightened.

通知チャネルの設計は、アシスタント自身が使うのと同じメッセンジャーです。運用者の携帯がそのままポケベルになり、「verify 失敗率が上昇中」のようなアラートがプッシュ通知で届きます。しきい値をこのノートにあえて書かないのは意図的です。紙の上でしきい値をでっち上げることは、このプロジェクトが一貫して避けてきた「測っていない主張」そのものだからです。しきい値は計測期間に観測されたベースラインから定め、その後で絞り込みます。

Layer 2 — the dashboard: seeing at a glance第2層 — ダッシュボード：一目で見る

design One screen, answerable in five seconds: is it green or red right now? The designed panel set: live status; a feed of recent task successes and failures; verify pass rate; error and retry rates; LLM calls per task (the loop detector); and latency p50/p95. An anomaly should be visible as a line falling or a tile turning red — before anyone writes a query. The dashboard reads the same records as the alerts; it is the alert rules made continuously visible.

design 一枚の画面で、5秒で答えられること — いま緑か、赤か。設計上のパネル構成は、ライブ状態、直近タスクの成功・失敗フィード、verify 通過率、エラー率と再試行率、タスクあたりの LLM 呼び出し数（ループ検知器）、レイテンシ p50/p95 です。異常は、誰かがクエリを書くより先に、線が落ちる・タイルが赤くなるという形で見えるべきです。ダッシュボードはアラートと同じ記録を読みます — アラートルールを常時可視化したものです。

Layer 3 — drill-down: asking why第3層 — ドリルダウン：なぜかを問う

built This is the one layer whose substrate already ships in the product. Every task persists its step-by-step timeline; the inspector drills from an intent to the exact key sequence sent; the console renders the screen states the run produced; and every approval decision lands in an append-only audit log. design What the operations layer would add on top: clicking a failure in the dashboard's feed and landing directly on the full trace — retrieval, plan, keys sent, the assertion that failed, the screen that came back — so that "it broke" becomes "the screen returned X where the plan expected Y, here." The aggregation queries and the dashboard-to-trace navigation are the unbuilt part; the trace itself is recorded today.

built この層だけは、その基盤がすでに製品として出荷されています。すべてのタスクはステップ単位のタイムラインを永続化し、インスペクタは意図から実際に送られたキー列までを掘り下げ、コンソールは実行が生んだ画面状態を表示し、すべての承認の決定は追記専用の監査ログに残ります。design 運用層がその上に足すのは、ダッシュボードのフィードで失敗をクリックすると完全なトレース — 検索・計画・送出キー・失敗した検証・返ってきた画面 — に直接着地する動線です。「壊れた」を「計画は Y を期待したが画面は X を返した、ここで」に変えるためのものです。未構築なのは集計クエリとダッシュボードからトレースへの導線で、トレース自体は今日すでに記録されています。

SLOs — which signals would carry objectivesSLO — どの信号に目標を載せるか

design The signals that would carry service-level objectives are the ones the funnel already watches: verify pass rate, the rate of executions that complete without needing human rescue, replan frequency, and plan latency at p95. Two honesty rules govern them. First, no target is stated in this note — the eval harness's published figures are measurements of a fixed 24-case suite, not operational baselines, and borrowing them as SLO targets would conflate the two. Targets come from a baseline period in real operation. Second, a verify failure spends error budget even when recovery succeeds: the system noticing its own mistake is better than not noticing, but the objective is to need the recovery less.

design SLO を載せる信号は、漏斗がすでに見ているものと同じです — verify 通過率、人の救援なしで完了した実行の割合、再計画の頻度、計画レイテンシの p95。これらを律する正直さのルールが2つあります。第一に、本ノートには目標値を書きません。評価ハーネスの公表数値は固定された24ケースのスイートに対する測定であって運用ベースラインではなく、それを SLO 目標に流用するのは両者の混同です。目標値は実運用のベースライン期間から定めます。第二に、リカバリが成功しても verify の失敗はエラーバジェットを消費します。自分の誤りに気づくシステムは気づかないものよりましですが、目標はリカバリの必要自体を減らすことだからです。

On-call — who gets called, with what in handオンコール — 誰が、何を手にして呼ばれるか

design The honest on-call design for a single-maintainer system is one person: the maintainer is the rotation. What makes that survivable is the funnel — the design goal is that an alert never arrives naked. It carries the failing signal, and the diagnosis is one click deep, because layer 3 already holds the trace. Escalation in the usual sense (a second responder) is a non-goal at this scale; the meaningful escalation path already exists in the product itself, where any execution the system cannot defend lands on a human's desk as a re-entry or needs-investigation outcome rather than being retried into the legacy system.

design 保守者が一人のシステムにおける正直なオンコール設計は、一人 — 保守者自身が当番表です。それを成り立たせるのが漏斗です。設計目標は、アラートが裸で届かないこと。アラートは異常を起こした信号を携えて届き、第3層がすでにトレースを保持しているため、診断は1クリック先にあります。通常の意味でのエスカレーション（第二応答者）はこの規模では目標にしません。意味のあるエスカレーション経路はすでに製品の中にあります — システムが正当化できない実行は、レガシーシステムへ再試行されるのではなく、再入力または要調査の結果として人の机に届きます。

Automatic rollback — two different things by the same name自動ロールバック — 同じ名前の異なる二つ

built Inside a single execution, rollback already exists and runs on the product path: a failed screen assertion triggers replanning up to a hard cap, then rolls the session back to a safe screen state instead of leaving a half-entered application behind; bad input data short-circuits straight to a person; and idempotency keys make a re-submission yield one application, not two. design What this note designs is rollback at the operations level — above executions, over the knowledge the system accumulates. The canonical case is a personal correction that turns out to be wrong: the designed mechanism rolls the correction set back to its previous version — the versioned, supersedes-chained storage this requires already ships — and quarantines the offending entry for human review rather than deleting it silently.

built 単一の実行の内側では、ロールバックはすでに存在し、製品の経路で動いています。画面検証の失敗は上限つきの再計画をトリガーし、それでも駄目ならセッションを安全な画面状態へ巻き戻し、入力途中の申請を残しません。不正な入力データは人へ直行し、冪等キーにより再送信しても申請は2件ではなく1件になります。design 本ノートが設計するのは、その上の運用レベルのロールバック — 実行の上位、システムが蓄積する知識に対するものです。典型例は、誤りだと判明した個人修正です。設計上の機構は、修正セットを前のバージョンへ巻き戻し（これに必要なバージョン管理と supersedes 連鎖つきの保存はすでに出荷済み）、問題のエントリを黙って消すのではなく隔離して人のレビューに回します。

育成 hygiene — detecting contamination育成の衛生 — 汚染の検知

design The growth loop is the product's most distinctive surface and its most distinctive risk: a wrong correction does not crash anything — it quietly bends every future plan for that operator. The boundary that limits the blast radius is already enforced in code (a correction cannot touch what the request grounds; that invariant measures 1.0 and is re-verified on every eval run). What remains is detecting contamination inside the territory corrections legitimately govern. The designed mechanism couples the growth loop to the eval harness: corrections enter versioned, as they already do; the eval suite re-runs on a schedule and after correction changes; and a metric regression that coincides with a correction version becomes a flagged, quarantinable event — fall back to the prior version, surface the correction for review. True unlearning (removing a correction's influence entirely) is listed in the product's stretch scope and is unbuilt; until it exists, quarantine-and-fallback is the designed containment.

design 育成ループはこの製品の最も特徴的な面であり、同時に最も特徴的なリスクです。誤った修正は何もクラッシュさせません — そのオペレーターの将来のすべての計画を静かに歪めます。被害半径を限定する境界はすでにコードで強制されています（修正は依頼が根拠づける値に触れられない。この不変条件の測定値は 1.0 で、評価実行のたびに再検証されます）。残る課題は、修正が正当に支配する領域の内側での汚染の検知です。設計上の機構は育成ループを評価ハーネスに結合します。修正は現状どおりバージョン付きで入り、評価スイートは定期と修正変更後に再実行され、修正バージョンと時を同じくする指標の後退は、フラグ付きの隔離可能なイベントになります — 前バージョンへフォールバックし、当該修正をレビューに浮上させる。真の unlearn（修正の影響の完全な除去）は製品のストレッチ範囲に記録されており未構築です。それが存在するまでは、隔離とフォールバックが設計上の封じ込めです。

The daily loop日々のループ

design The designed steady state, by trigger rather than by clock: the dashboard and alerts stand watch continuously. A guideline change triggers an eval regression run before it is trusted. A change on the legacy side announces itself as verify failures — the adapter is fixed, not the symptom. A model update means the full eval suite re-runs before the new model serves. And growth hygiene is periodic: corrections are reviewed and pruned, because the previous section's detector only catches what regresses a metric, not what merely accumulates.

design 設計上の定常運転は、時刻ではなくトリガーで回ります。ダッシュボードとアラートは常時の見張り。指針の変更は、信頼する前に評価回帰の実行をトリガーします。レガシー側の変更は verify の失敗として自己申告してくるので、直すのは症状ではなくアダプタです。モデルの更新は、新モデルが本番に就く前に評価スイートの全再実行を意味します。そして育成の衛生は定期作業です — 前節の検知器が捕まえるのは指標を後退させるものだけで、ただ溜まっていくものは捕まえないため、修正は定期的にレビューし、剪定します。

One instrumentation point, two cost regimes一つの計測点、二つのコスト体系

design The wrapper that instruments the LLM client is designed provider-agnostic, because the numbers worth watching change with the serving path: an on-premise deployment watches GPU utilization, throughput and VRAM headroom, while an API-served deployment watches tokens and spend. Only the cost arithmetic differs per path; the instrumentation point, the usage table, and the alert rules above it stay the same.

design LLM クライアントを計測するラッパーはプロバイダ非依存に設計します。見るべき数値が配信経路で変わるからです — オンプレミス配備では GPU 使用率・スループット・VRAM の余裕を、API 配信ではトークンと支出を見ます。経路ごとに異なるのはコストの算術だけで、計測点・利用量テーブル・その上のアラートルールは同一のままです。
