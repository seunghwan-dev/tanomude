Productization — a design exercise製品化構想（design intent）

Operate your legacy core systems in natural language — without touching them.基幹系を、触らずに、自然言語で。

Tanomude is a non-invasive, on-premise AI layer placed over your existing core systems — bridging legacy to modern without modifying either. Two layers: operation and judgment. It runs inside your closed network, and every result is verified.

既存の基幹システムには一切手を入れない。Tanomude は、レガシーとモダンのあいだに置く非侵襲のオンプレミス AI レイヤーです。構成は「操作」と「判断」の二層。閉域ネットワークの中で動作し、すべての結果は検証を通ります。

01

The problem — knowledge succession and approval lead-time問題 — ナレッジ継承と、承認リードタイム

Tacit knowledge is breaking down in two places at once.

暗黙知の断絶は、二か所で同時に起きています。

First, legacy operation: the know-how to run the core system lives only in the heads of veterans approaching retirement.

一つ目はレガシー操作。基幹系を動かすノウハウは、定年を控えたベテランの頭の中にしか存在しません。

Second, approval criteria: what passes and what gets rejected exists only in the section manager's (課長's) head. New staff can only guess, rejections repeat, and approval lead-time balloons.

二つ目は承認基準。何が通り、何が却下されるかは課長の頭の中にしかありません。新人は推測で入力するしかなく、却下が繰り返され、承認リードタイムが膨らんでいきます。

And yet manufacturing will not replace its closed legacy systems outright — the desire to own, the sunk cost, the closed network. Since "rebuild" is off the table, what is needed is not a replacement but a knowledge-succession (ナレッジ継承) layer on top of what already exists.

それでも製造業は、閉域のレガシーをリプレースしません。自社で所有していたいという意志、積み上がったサンクコスト、そして閉域ネットワークという前提。「作り直す」が選択肢にならない以上、必要なのはシステムの置き換えではなく、ナレッジ継承の層を既存システムの上に重ねることです。

02

Two layers — operation and judgment二つのレイヤー — 操作と判断

The operation layer. It operates the core system in natural language through the existing interface. No modification, no integration work on the customer side.

操作レイヤー。既存の端末をそのまま使い、自然言語で基幹系を操作します。顧客側のシステム改修も、連携開発も不要です。

The judgment layer. It encodes the manager's approval criteria and enforces them at input time. The goal is first-pass approval: eliminating rework.

判断レイヤー。課長の承認基準をコード化し、入力の時点で適用します。狙いは一発承認、すなわち手戻りをなくすことです。

The capture mechanism that is actually built is 育成 (growth): the system learns criteria from the manager's real rejections and corrections (revealed preference; no one has to write a spec). The cold-start guideline source is defined as a pluggable upstream port — a defined seam, deliberately out of scope.

基準の獲得メカニズムとして実装済みなのは「育成」です。課長が実際に行った却下・修正から基準を学習します(顕示選好。仕様書を書いてもらう必要はありません)。コールドスタート時のガイドライン供給源は、差し替え可能な上流ポートとして定義済みです(意図的にスコープ外とした、定義済みのシームです)。

HONESTY正直に

The manager-guideline channel is not yet in production; 育成 is the production path.

マネージャー・ガイドラインのチャネルは、現時点では本番未実装です。本番経路は育成です。

03

Differentiation — non-invasive, without replacement差別化 — 置き換えない、非侵襲という立ち位置

Modern backoffice SaaS replaces the employee-facing screen and connects to existing systems via REST API, in the cloud. It is an excellent choice where systems expose APIs and the company will adopt a new UI. Tanomude addresses the cases where those premises do not hold.

モダンな業務 SaaS は、従業員が使う画面そのものを置き換え、REST API で既存システムに接続します(クラウド)。API が公開され、新しい UI の導入を受け入れられる企業には優れた選択肢です。Tanomude は、その前提が成り立たない領域を対象とします。

"No new screen, no system change. We operate what you already have, under human approval."

「画面も、システムも、置き換えない。既存のものを、承認を介して操作する。」

A market for learning approval criteria already exists. Tanomude focuses on the on-prem, legacy, in-house-criteria segment — a different space, not a head-on contest.

承認基準を学習する市場は、すでに存在します。Tanomude が対象とするのは、オンプレ × レガシー × 自社基準という領域です。正面から競うのではなく、対象が異なります。

04

Why this is defensible — not RPA; trust as codeなぜ守れるのか — RPA ではなく、「信頼」をコードにする

RPA is deterministic record-and-replay; it cannot judge. Tanomude targets exactly the territory RPA leaves to humans.

RPA は決定的な記録・再生であり、判断はできません。Tanomude が狙うのは、まさに RPA が人間に残した領域です。

The defensibility is threefold. First, the verify gate: boundary_respect is a code-enforced invariant, 0.25→1.0. Second, the eval harness: behavior is measured across 4 datasets, and the effect of 育成 is measured as growth_delta. Third, 育成 accumulation: customer-specific criteria pile up with use and become switching cost. These are what the harness measures, not marketing wins.

守りの源泉は三つあります。一つ目は検証ゲート。boundary_respect はコードで強制される不変条件で、0.25→1.0 です。二つ目は評価ハーネス。4 つのデータセットで挙動を測定し、育成の効果は growth_delta として計測します。三つ目は育成の蓄積。使うほど顧客固有の基準が積み上がり、スイッチングコストになります。いずれも「ハーネスが何を測っているか」であって、マーケティング上の戦果ではありません。

Anthropic's own Computer Use guidance says to use human-in-the-loop for consequential actions, and not to use raw computer-use for accuracy-critical or sensitive work. A core system is exactly that. Tanomude is the trust wrapper that makes computer-use safe for a core system.

Anthropic 自身の Computer Use ガイダンスは、重大なアクションには HITL(人間の承認)を置くこと、精度が重要な業務・機微な業務には生の computer-use を使わないことを明確に述べています。基幹系はその典型です。Tanomude は、computer-use を基幹システムに対して安全にする信頼ラッパーです。

05

Local-first feasibilityローカルファーストの実現性

As of 2025, external benchmarks show: on the GUI-grounding benchmark ScreenSpot-Pro, the open local model UI-TARS-1.5-7B scores 61.6% versus Claude's 27.7% — an open LOCAL model beats cloud on GUI grounding. It runs on a single consumer GPU (~24GB; ~4–5GB quantized), in a one-appliance-to-N-thin-clients configuration — no per-person GPU.

外部ベンチマーク(2025年時点)では、GUI グラウンディングのベンチマーク ScreenSpot-Pro で、オープンなローカルモデル UI-TARS-1.5-7B は 61.6% を記録し、Claude の 27.7% を上回ります。「画面のどこを操作するか」という一点では、オープンなローカルモデルがクラウドに勝っています。コンシューマー GPU 一枚(約 24GB、量子化で約 4〜5GB)で動作し、アプライアンス 1 台に N 台のシンクライアントをぶら下げる構成です。一人一枚の GPU は要りません。

Reliability comes from OUT-OF-BAND artifact readback: we verify the persisted result, not the screenshot.

信頼性は、スクリーンショットの見た目ではなく、帯域外のアーティファクト読み戻しで担保します。永続化された結果そのものを検証するのです。

HONESTY正直に

Autonomous long-horizon operation is still weak for open models — which is exactly why the HITL + verify + 育成 harness exists. The harness, not the model, carries the reliability.

オープンモデルの自律的な長時間運転は、まだ弱いです。だからこそ HITL + 検証 + 育成のハーネスがあります。信頼性を担うのはモデルではなく、ハーネスです。

07

Scaling economicsスケーリングの経済性

Design-validated, not yet built設計検証済み・未実装

Declarative WorkflowDef: roughly 190 lines of workflow-specific code, of which about 85% is expressible as data — within the existing primitives; the honest caveat is that novel transforms and branches still need code. Today, adding a workflow means ~350–600 LOC across 6+ files; with WorkflowDef it becomes ONE data file and near-zero code.

宣言的 WorkflowDef。ワークフロー固有のコードは約 190 行で、その約 85% はデータとして表現できます(既存プリミティブの範囲内。新規の変換や分岐には依然コードが必要、というのが正直な但し書きです)。現状、新しいワークフローの追加は 6 ファイル以上・約 350〜600 行の変更ですが、WorkflowDef ではデータファイル 1 つ+ほぼゼロのコードになります。

A new target SYSTEM irreducibly needs a code adapter (~200–600 LOC per tier). Adapter tiering — tn5250 / DOM / vision — is a minimal ABC extension; the core loop stays untouched.

新しい対象システムへの対応は、本質的にコードアダプタを必要とします(ティアあたり約 200〜600 行)。アダプタは tn5250 / DOM / vision の 3 ティアで、最小限の ABC 拡張として実装し、コアループには手を入れません。

This is what makes subscription and multi-workflow expansion credible. Labeled clearly: design-validated, not yet built.

これが、サブスクリプションと複数ワークフロー展開を成立させる根拠です。ラベルは明確です。設計段階で検証済み、未実装。

10

What's nextこれから — 次の一手

Tanomude is built to keep operating & evolving. The concrete next engineering steps:

Tanomude は「運用しながら進化する」プロダクトです。具体的な次の一手は以下のとおりです。

  • Promote the request's purpose to a discrete form field with a deterministic pin, hardening the explicit-input boundary.
  • Per-field re-entry guidance driven by the validation codes.
  • Ingest real operating manuals and measure "a correction beats the manual rule" on them.
  • A continuous-integration smoke test of the full container stack.
  • Promote the model-scored eval suite into a CI regression gate (today it runs locally; CI gates only the model-free subset).
  • 依頼の目的を独立したフォーム欄に昇格し、決定論的に固定して、明示入力の境界を強化する。
  • 検証コードに基づく、フィールド単位の再入力ガイドメッセージ。
  • 実際の手順書を取り込み、「修正が手順書ルールに勝つ」ことを測定する。
  • コンテナスタック全体の継続的インテグレーション・スモークテスト。
  • モデル採点の評価スイートを CI 回帰ゲートへ昇格する（現在はローカル実行で、CI はモデル非依存サブセットのみをゲート）。
